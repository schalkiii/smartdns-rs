项目：smartdns-rs · 诊断日期：2026-07-09 ~ 07-10 · 作者：AI 排查
结论先行：WebUI 的命中率计算公式没有错，44% 是真实值；真正拉低它的是 「否定缓存（negative caching）缺失」。原版 smartdns 把 NXDOMAIN/NODATA 也缓存起来， 而本分支 `dns_mw_cache.rs` 那段逻辑被注释掉了。修复已实现并通过编译，命中率预计可回到 90%+ 量级。

3. 流量构成：为什么否定查询这么多

否定应答里绝大多数是双栈域名的 AAAA 查询无 AAAA 记录（如 www.doubao.com AAAA → NXDOMAIN）。 原版 smartdns 按 SOA 的 TTL 缓存这些否定结果，重复查询直接命中；本分支每次都重新解析 → 全计为 miss。 这正是 44% vs 90% 的差距来源。

4. 已实施的修复

4.1 src/dns_mw_cache.rs

// 新增：对 NXDOMAIN / NODATA 响应缓存（否定缓存）
fn insert_negative(&self, query, group, res) {
    let ttl = res.soa_minimum_ttl().clamp(60, 3600);   // 限幅 60s~3600s
    self.cache.insert_records_negative(query, group, ttl, Instant::now()).await;
}
// handle() 的 Ok(lookup) 分支：在原有正向缓存之前，先判断是否否定应答
//   - NXDOMAIN / NODATA  → insert_negative()
//   - SERVFAIL / REFUSED → 不缓存（保持原行为）
// get() 命中否定条目 → 原样返回 + mark_from_cache() + query_hits+1
//   否定条目不进 prefetch_heap（无需刷新）

5. 关于「600s TTL 硬上限」是否有必要增大

结论：在现有 serve-expired yes 配置下，对命中率而言没有必要增大；但作为「减少上游预取抖动」的可选优化，建议后续补一个 cache-max-ttl 配置项。

现状：insert 把每条缓存条目 TTL 硬封顶 min(600, record_ttl)。原版 smartdns 尊重记录自然 TTL（常达数小时）。

  • 命中率角度：过期条目由 serve-expired（7 天窗口）兜底返回，实测 83% 仍命中（见 §2）。所以 600s 上限不影响命中率，本次不动它。
  • 上游负载角度：600s 上限让条目比原版更早过期 → 更频繁触发后台 prefetch 刷新（实测 bg_total_queries=5443），增加上游查询与 CPU 抖动。
  • 建议：若想更贴近原版、减少预取抖动，应补 cache-max-ttl 配置（默认 0 = 用记录自然 TTL），而非写死更大魔数。已记入后续待办，本次不纳入（保持改动聚焦、避免重新构建验证范围扩大）。
