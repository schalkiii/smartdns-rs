2. 代码 Review 发现的潜在性能问题（按严重度排序）

P0 上游无熔断 / 健康检查机制

dns_client.rs · NameServerGroup::lookup (L447-493) 与 NameServer::lookup (L597-689)

现象：NameServerGroup::lookup 对 self.servers 里的每一个上游都 tokio::spawn 一个 task，没有任何"已知故障跳过"逻辑。NameServer 内部也没有失败计数。

后果：每个客户端查询 + 每次 prefetch 都会向所有上游（含死的）发请求。对死/慢上游，每个查询都要空耗满 NS_QUERY_TIMEOUT(3s)、且全程持有 1 个信号量 permit，并产生一条 debug 日志。
实测佐证（当前日志）：lookup timed out = 3310 次，resource too busy = 2606 次，且连原本健康的 doh.pub/dns.alidns.com 也出现间歇超时。

✅ 已实现（2026-07-08，待编译验证）：新增 fail_streak: AtomicU32 + last_failure: Mutex<Option<Instant>>。连续失败 ≥ NS_FAIL_THRESHOLD(3) 次进入 NS_COOLDOWN(30s) 冷却，期间 lookup 直接进入 cooldown 分支跳过该上游；成功即 record_success() 清零；超时/非重试错误触发 record_failure()。死上游不再每查询空耗 3s + 刷日志 + 占 permit。

P1 缓存 prefetch 扫描是 O(cache_size) 且持全局锁

dns_mw_cache.rs · get_expired (L696-744) 与 start_prefetching 调度 (L136-219)

现象：get_expired 用 for (query, entry) in cache.iter_mut() 遍历整个 LruCache，并在 cache.lock().await 持锁下进行。start_prefetching 在任意缓存记录的 TTL 到期时（L398-402 notify_after(ttl)）就触发一次全扫描。

后果：cache-size 65536 下，每次 prefetch 扫描最多扫 6.5 万条、且持有所有查询都要抢的同一把锁 → 查询延迟尖刺 + 无意义 CPU。缓存越热、域名越多，扫描越频繁。

✅ 已实现（2026-07-08，待编译验证）：新增 prefetch_heap: Mutex<BinaryHeap<Reverse<PrefetchKey>>>（仅按 Instant 排序）。insert 时压入 (valid_until, query)；get_expired 改为只 peek/pop 已过期的堆顶条目（O(到期数)），不再遍历整张 LruCache。L177 调试用的 cache.lock().await.len() 已改为 entry_count().await（非全扫描）。

P2 缓存使用单一全局 Mutex<LruCache>

dns_mw_cache.rs · L472（定义），L527/L655（get/insert 加锁）

现象：所有缓存读/写都串行在同一把 tokio Mutex 上。

后果：家庭网络 QPS 下无感；高 QPS 时成为天花板。

✅ 已实现（2026-07-08，待编译验证）：DnsCache 改为 shards: Vec<Arc<Mutex<LruCache>>>（CACHE_SHARD_COUNT=16），按 shard_index_of(query) 稳定路由；get/insert/clear/cached_records/snapshot_entries 均按分片加锁。就绪堆为单一结构（prefetch 本就是单 worker）。落盘接口改为 persist_to/load_from（启动时重建堆）。

P3 dualstack-ip-selection yes 与 speed-check-mode none 自相矛盾

dns_mw_dualstack.rs · which_faster (L146-173)；dns_mw_ns.rs · L250-266

现象：speed-check-mode none 时，dns_mw_ns 会强制 response_strategy = FastestResponse，且 multi_mode_ping_fastest 因无 ping 地址返回 None；which_faster 落入 this_ping None => that_faster = true 分支，偏向 AAAA。

后果：双栈选择实际无法测速，且逻辑上偏向 AAAA。当前因 force-aaaa-soa yes 让 AAAA 秒回 SOA、无真实 AAAA 可比较，故基本无害——但配置语义矛盾、易误导。

✅ 已修复（2026-07-08）：已从 smartdns.conf 删除 dualstack-ip-selection yes 及其 threshold 行（与 speed-check-mode none 矛盾，属空操作）。

P4 死上游超时期间持有 permit（由 P0 顺带解决）

dns_client.rs · L646 + L649

semaphore.acquire() 拿到 permit 后，在 timeout(3s) 等待期间一直持有。死上游会占满其 1/12 的并发槽位达 3s；持续负载下每波最多 12 个后台 task 挂死。因上限 12，不影响客户端延迟，但浪费且会触发 resource too busy。P0 的熔断会让死上游被跳过，此问题自然消失。

P5 NameServerGroup 总是 fan-out 到全部上游（设计如此，P0 缓解）

fastest-response 需要多个响应来挑最快，全打是预期行为。P0 熔断后死的会被跳过；已知慢的（如 reitmeier ~2s）仍每次都打，可接受。

3. 运维层发现（非代码 bug，但直接影响性能）

运维 同时运行了两个 smartdns 实例，共享端口与缓存文件

当前进程：PID 2260（07-07 23:52:54 启动，我昨晚后台遗留）+ PID 96096（08-08 08:25:26 启动，你今早启动）

两者都 LISTENING 在 9053 / 8083（smartdns 启用 SO_REUSEPORT 允许端口复用）。后果：

  • 查询被分流到两个进程，行为不一致、难排查；
  • prefetch 双倍打上游 → 上游查询量翻倍 → 更多 3s 超时（这正是"健康上游也开始超时"的放大器）；
  • 共享同一 ./data/smartdns.cache → 缓存文件损坏风险；日志互相交错。

建议：系统里只保留一个 smartdns 实例。最干净的做法是杀掉昨晚遗留的 PID 2260，保留你今早启动的 96096（二者都是带 web-ui 的同一 exe）。需要我执行请确认（属于停止进程操作）。
